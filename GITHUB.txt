# Git & GitHub Workflow Guide (End-to-End)

Author: Mounika
Purpose: Maintain a clear workflow for creating repositories, pushing code, handling `.gitignore`, and resolving common Git issues.

---

# 1. Creating a Repository on GitHub

1. Go to **GitHub → New Repository**
2. Enter:

   * Repository Name
   * Description (optional)
3. Choose:

   * Public / Private
4. Select:

   * Add README (optional)
   * Add .gitignore (optional)

Click **Create Repository**

Example Repository URL:

```
https://github.com/username/repository-name.git
```

---

# 2. Setting Up Local Project

Open terminal or VS Code terminal.

Navigate to project directory:

```
cd D:\ProjectFolder
```

Initialize Git:

```
git init
```

Verify repository initialized:

```
git status
```

You should see:

```
On branch main
No commits yet
```

---

# 3. Connecting Local Repository to GitHub

Add remote repository:

```
git remote add origin https://github.com/username/repository-name.git
```

Verify remote:

```
git remote -v
```

Expected output:

```
origin https://github.com/username/repository-name.git (fetch)
origin https://github.com/username/repository-name.git (push)
```

---

# 4. Adding Files to Git

Check project files:

```
git status
```

Add files:

```
git add .
```

or specific file:

```
git add filename
```

Commit files:

```
git commit -m "Initial project commit"
```

Push code:

```
git push -u origin main
```

---

# 5. Standard Daily Workflow

Whenever you modify code:

### Step 1 – Check status

```
git status
```

### Step 2 – Add changes

```
git add .
```

### Step 3 – Commit

```
git commit -m "Describe changes"
```

### Step 4 – Push

```
git push
```

# 6. Creating `.gitignore`

`.gitignore` prevents unnecessary files from being uploaded to GitHub.

Example `.gitignore` for Python:

```
venv/
__pycache__/
*.pyc
.env
data/
```

If ignoring only specific files:

```
data/Expenses.json
```

If ignoring entire folder:

```
data/
```

---

```
Assignments

 │   ├─ data
 │   ├─ venv
 │   └─ .gitignore
 │
```

Example `module1/.gitignore`

```
venv/

```

Example `module2/.gitignore`

```
venv/

```

# 8. Checking If Files Are Ignored

Show ignored files:

```
git status --ignored
```

Check specific file:

```
git check-ignore -v module1/data/Expenses.json
```

---

# 9. If File Was Already Committed Before `.gitignore`

`.gitignore` works only for **untracked files**.

If file already tracked:

Remove it from Git tracking:

```
git rm --cached module1/data/Expenses.json
```

Commit change:

```
git commit -m "Remove tracked data file"
```

Push:

```
git push
```

File remains **locally but removed from GitHub tracking**.

---

# 10. Error: "fatal: not a git repository"

Cause:

Git not initialized in the folder.

Solution:

Navigate to correct folder:

```
cd project-folder
```

Initialize repository:

```
git init
```

---

# 11. Error: "pathspec did not match any files"

Example error:

```
fatal: pathspec 'module2/.gitignore' did not match any files
```

Cause:

File does not exist.

Solution:

Create the file first:

```
module2/.gitignore
```

Then run:

```
git add module2/.gitignore
```

---

# 12. Error: "Updates were rejected because the remote contains work"

Example error:

```
! [rejected] main -> main (fetch first)
```

Cause:

GitHub repository already contains commits (README, .gitignore).

Solution:

Pull remote changes first:

```
git pull origin main --rebase
```

Then push:

```
git push origin main
```

# 15. Best Practices

Always ignore:

```
venv/
__pycache__/
.env
*.pyc
```

Never upload:

* API keys
* Large datasets
* Virtual environments

Always upload:

* Source code
* Documentation
* Requirements file

Example clean repo:

```
project
│
├── src
├── README.md
├── requirements.txt
└── .gitignore
```

Each module should contain:

```

src/
data/
.gitignore
```

```
# Module 2

This module contains NLP experiments including tokenization and preprocessing.

```

---

# 17. Final Workflow Summary

1. Create repository on GitHub
2. Clone or initialize repository
3. Add project files
4. Create `.gitignore`
5. Commit code
6. Push to GitHub
7. Use `git status` frequently
8. Resolve conflicts using `git pull`
